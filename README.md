# 🏦 AI Bank Consultant: Agentic RAG-система

## 📌 Описание проекта
Данный проект представляет собой разработку вопросно-ответной системы (RAG — Retrieval-Augmented Generation) для консультирования клиентов банка. Система способна анализировать банковскую документацию и предоставлять точные ответы, опираясь исключительно на факты из базы знаний. Текущий RAG-пайплайн спроектирован как базовое промежуточное звено для дальнейшего перехода к полноценной агентной архитектуре (Agentic RAG).

## 🛠 Технологический стек
* **Оркестрация и Агентный подход:** LangChain
* **Векторная база данных:** FAISS (in-memory)
* **Модель эмбеддингов:** `intfloat/multilingual-e5-large` (выбрана за высокое качество работы с русским языком)
* **Генеративная модель (LLM):** `Qwen/Qwen2.5-1.5B-Instruct`
* **Дополнительно:** Rank-BM25 (для гибридного поиска)

## 📂 Структура репозитория
* `/NLP_II/` — синтетический набор данных из 5 документов (условия кредитования, вклады, ипотека, требования к заемщикам, FAQ).
* `NLP_II_Project_Kozhenkova.ipynb` — Jupyter Notebook с полным пайплайном данных, ретривера и LLM.
* `README.md` — описание проекта и выводы.

## 📊 Этапы реализации и архитектурные выводы

### Этап 1. Подготовка данных и эмбеддинги
Документы были очищены и приведены к единому формату. В ходе сравнения стратегий чанкинга выбран `RecursiveCharacterTextSplitter`. Доказано, что он оптимально сохраняет логическую структуру документов, не разрывая пункты маркированных списков. Векторизация выполнена с помощью модели `multilingual-e5-large`.

### Этап 2. Система ретрива (Поиск)
Был реализован и протестирован кастомный пайплайн поиска:
1. **Сравнение алгоритмов:** A/B тестирование показало, что Базовый векторный поиск (Similarity) превосходит алгоритм MMR для точных финансовых запросов.
2. **Контекстное сжатие:** Внедрен фильтр на основе L2-дистанции (строгий порог отсечения `0.305`), который безжалостно отсекает информационный шум.
3. **Метрики качества:** На синтетическом датасете из 20 вопросов система показала высочайшую точность:
   * **Hit Rate@3:** 0.95 (95% запросов находят нужный файл)
   * **MRR:** 0.892

### Этап 3. Интеграция с LLM и Анализ уязвимостей
В систему интегрирована локальная модель Qwen 1.5B. Реализован жесткий системный промпт (temperature=0.1) и память диалога (Memory).
**Edge Case (Утечка контекста):** В ходе тестирования выявлен эффект галлюцинации из-за склейки контекста. При резкой смене темы (от ипотеки к бизнес-кредитам) ретривер подтянул условия ипотеки, а модель скомбинировала запрос про бизнес с ипотечным лимитом в 30 лет. Это доказывает, что базовую конкатенацию истории необходимо заменять на более сложные паттерны (LLM-Router или Query Rewriting).

### Этап 4. Оптимизация производительности
Внедрение in-memory кеширования продемонстрировало феноменальную вычислительную эффективность. Время генерации ответа LLM «с нуля» составило ~6.59 сек, в то время как выдача готового ответа из кеша заняла всего 0.00013 сек. Итоговое **ускорение системы составило более 49 000 раз**, что позволяет масштабировать решение для работы под высокой нагрузкой без затрат на дорогую GPU-инфраструктуру.

## 🚀 Вектор развития (Agentic Architecture)
Текущий RAG-пайплайн обернут в функцию, готовую к использованию в качестве инструмента (`Tool`) для LangChain Agent. Следующим этапом планируется внедрение агента-маршрутизатора (Router Agent), который будет решать:
1. Искать ответ в локальной базе FAISS (текущая реализация).
2. Вызывать инструмент "Калькулятор" для расчета платежей.
3. Переключать клиента на оператора.
